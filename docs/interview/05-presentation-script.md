# RepoOps - 面试讲述脚本（3-5 分钟版）

## 开场（30秒）

"我做了一个叫 RepoOps 的项目，它是一个 Agent 工作流框架，能把 GitHub Issue 自动变成经过测试验证的 PR。整个流程是：读 Issue → 扫描仓库 → LLM 生成计划 → LLM 生成代码编辑 → 应用到文件 → 跑测试验证。如果测试没过，会自动回滚、把失败信息喂回 LLM 重试。"

## 为什么做（30秒）

"动机是：现在的 AI 编程工具擅长代码补全，但缺少端到端的闭环。我想探索 Agent 能不能把从需求理解到代码验证的整个流程自动化，同时保证可靠性——不只是能生成代码，还要能验证代码是对的。"

## 怎么做的（1分钟）

"技术上有三个关键决策：

第一，用 LangChain 加 Pydantic 做结构化输出。LLM 返回严格的 JSON，不是自由文本。Pydantic 的 schema 直接注入 prompt 里告诉 LLM 该返回什么格式。

第二，代码修改用精确字符串匹配。LLM 返回原始代码片段和替换代码，系统做 str.replace。这个方案语言无关、实现简单，准确性的问题通过 prompt 工程解决。

第三，闭环重试。测试没过的话，自动回滚文件，把测试失败输出喂回 LLM 重新生成编辑。不是简单 retry，而是每次重试都带着失败的上下文。"

## 踩坑故事（1.5分钟，重点）

"开发过程中最有意思的问题是 snippet 匹配失败。

用 Gemini 跑端到端测试时，编辑全部静默失败了。我排查发现，Gemini 返回的原始代码片段和文件实际内容有微妙差异——比如函数签名里丢了类型注解，`def divide(a, b)` 而文件里是 `def divide(a: float, b: float) -> float`，还跳过了整个 docstring。

面对这个问题有两个方向：一是做模糊匹配，容忍 LLM 的不精确；二是改 prompt，从源头约束 LLM。

我选了改 prompt。加了 CRITICAL RULES，明确告诉 LLM：original_snippet 必须从文件内容逐字复制，包括类型注解、docstring、空格，一个字符都不能错，否则精确匹配会失败。

结果：匹配成功率从 0% 到了 100%。而且这个方案比模糊匹配更简单。

这件事教会我一个原则：**不要用工程复杂度去补偿 LLM 的行为，而要用 prompt 去约束 LLM 的行为**。"

## 收尾（30秒）

"目前在 demo 仓库上跑通了完整流程：自动修复 divide-by-zero bug，5 个测试全部通过，输出 plan.json、patch.diff、pr_draft.md 等完整制品。

这个项目让我深刻体会到：Agent 系统的可靠性不来自更强的模型，而来自更好的工程设计——结构化输出、闭环验证、回滚机制，这些传统软件工程的手段在 Agent 系统里同样关键。"

## 时间分配建议

| 部分 | 时间 | 重点 |
|------|------|------|
| 开场 + 动机 | 1 min | 快速建立上下文 |
| 技术决策 | 1 min | 展示工程判断力 |
| 踩坑故事 | 1.5 min | **最重要**，展示调试能力和决策过程 |
| 收尾 | 0.5 min | 提炼认知，留下深刻印象 |

## 注意事项

- 不要念代码，用白板或口头描述逻辑
- 踩坑故事要讲**决策过程**（为什么选 A 不选 B），不只是讲结果
- 如果面试官追问细节，引导到 03-problems-and-solutions.md 中的内容
- 准备好回答"如果要生产化怎么做"——这个问题几乎一定会被问到
